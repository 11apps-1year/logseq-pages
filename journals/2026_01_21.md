- Claude Code 해커톤에서 우승하고 나서 70가지 팁을 정리했어요.
  솔직히 처음엔 저도 Claude Code를 과소평가했거든요. 그냥 AI 코딩 도구겠지 했는데, 제대로 쓰는 법을 알고 나니까 완전히 달라지더라고요.
  이제부터 실전에서 바로 써먹을 수 있는 모든 걸 공유할게요.
- 1/ 먼저 마인드셋부터 바꿔야 해요. Claude Code는 시니어 개발자가 아니라 똑똑한 주니어 개발자예요.
  시니어한테는 "인증 기능 만들어줘"라고 하면 알아서 하잖아요? 근데 주니어한테는 "로그인 폼 만들고, JWT 생성하고, 리프레시 토큰 구현하고..." 이렇게 쪼개서 말해줘야 하거든요.
  Claude Code도 똑같아요. 작업을 작게 쪼갤수록 결과물이 훨씬 좋아지더라고요.
- 2/ 그래서 제가 발견한 게 Plan Mode예요. Shift+Tab 누르면 토글되는데요, 이게 진짜 게임체인저였어요.
  Plan Mode 켜고 "이커머스 사이트 만들어줘"라고 하면 먼저 전체 계획을 세워줘요. 그다음 각 단계를 하나씩 물어보면서 진행하거든요.
  반대로 YOLO Mode는 그냥 바로 실행해요. 간단한 작업엔 빠르지만, 복잡한 건 Plan Mode가 훨씬 안전해요.
- 3/ 여기서 핵심은 컨텍스트 관리예요. 이게 Claude Code 잘 쓰는 사람과 못 쓰는 사람을 나누는 결정적 차이거든요.
  컨텍스트는 신선한 우유라고 생각하면 돼요. 시간이 지나면 상해요. Claude Code도 대화가 길어지면 초반 내용을 까먹거나 헷갈리기 시작하거든요.
  제 경험상 3~5개 작업마다 컨텍스트를 리셋해주는 게 제일 좋더라고요.
- 4/ 컨텍스트 리셋하기 전에 꼭 해야 할 게 있어요. HANDOFF.md 파일 만들기예요.
  이 파일에는 "지금까지 뭐 했고, 다음에 뭐 해야 하고, 주의할 점은 뭔지" 적어두는 거예요. 그러면 새 세션 시작할 때 이 파일만 읽어주면 바로 이어서 작업할 수 있거든요.
  HANDOFF.md 없이 컨텍스트 날리면 진짜 멘붕 와요. 제가 한 번 그랬다가 2시간치 작업 흐름 설명하느라 시간 엄청 낭비했거든요.
- 5/ 컨텍스트 관리 명령어는 두 개예요. /compact랑 /clear인데요.
  /compact는 대화 내용을 요약해서 압축해줘요. 중요한 정보는 유지하면서 토큰만 줄이는 거죠. 작업 흐름이 끊기지 않아서 좋아요.
  /clear는 완전히 초기화예요. 깨끗하게 시작하고 싶을 때 쓰는데, HANDOFF.md 없으면 위험하거든요.
  저는 /compact 3번 쓰고 나면 /clear하고 새로 시작하는 패턴으로 써요.
- 6/ 근데 Claude Code는 200k 토큰까지 쓸 수 있잖아요? 그럼 계속 써도 되는 거 아니냐고요?
  이론적으론 그래요. 근데 실전에서는 토큰이 80~100k만 넘어가도 품질이 떨어지기 시작하더라고요. 응답이 느려지고, 앞뒤가 안 맞는 코드도 나오고요.
  그래서 저는 토큰 100k를 넘기 전에 무조건 리셋해요. 아깝게 느껴져도 이게 결국 시간 절약입니다.
- 7/ 이제 환경 설정 얘기 좀 할게요. 제일 먼저 해야 할 건 CLAUDE.md 파일 만들기예요.
  이 파일은 프로젝트 루트에 두는 일종의 설명서거든요. "이 프로젝트는 뭐고, 어떤 규칙으로 코딩하고, 파일 구조는 어떻고" 다 적어두는 거예요.
  Claude Code가 작업 시작할 때마다 이 파일을 먼저 읽어요. 그러면 매번 같은 설명 안 해도 되거든요. 진짜 엄청 편해져요.
- 8/ CLAUDE.md에는 이런 것들 넣으면 좋아요. 프로젝트 개요, 기술 스택, 코딩 컨벤션, 주요 파일 위치, 절대 건드리면 안 되는 파일 목록, 테스트 실행 방법...
  저는 여기에 "절대 console.log 남기지 마" 같은 규칙도 박아뒀어요. 그랬더니 Claude Code가 알아서 지켜주더라고요.
  처음 세팅하는 데 10분 걸리지만, 이후에 몇 시간씩 절약돼요.
- 9/ /init 명령어도 꼭 알아야 해요. 새 프로젝트 시작할 때 치면 기본 구조를 자동으로 잡아줘요.
  "React + TypeScript + Tailwind 프로젝트 init해줘" 이렇게 말하면 package.json, tsconfig.json, 폴더 구조까지 다 만들어주거든요.
  수동으로 하나씩 만드는 것보다 훨씬 빠르고 실수도 없어요. 저는 이제 모든 프로젝트를 /init으로 시작합니다.
- 10/ 터미널 쓰는 사람이면 alias 설정 꼭 하세요. 매번 claude 치기 귀찮잖아요.
  저는 .zshrc에 alias c="claude" 이렇게 넣어뒀어요. 그러면 c "할 일" 이렇게만 쳐도 바로 실행돼요.
  alias cc="claude code review" 이런 식으로 자주 쓰는 명령어 조합도 alias로 만들어두면 생산성이 엄청 올라가요.
- 11/ 음성 코딩 얘기도 해야죠. 저는 Wispr Flow 쓰는데요, 진짜 혁명이에요.
  키보드 치는 것보다 말하는 게 훨씬 빨라요. "Claude, 로그인 폼 만들어주는데 이메일이랑 비밀번호 validation 추가해줘" 이렇게 말하면 바로 실행되거든요.
  Superwhisper도 좋다고 들었어요. 둘 다 한글 지원되고, 정확도도 높아요. 타이핑 피곤하면 음성 코딩 진짜 추천드려요.
- 12/ 이제 MCP 얘기를 안 할 수가 없네요. Model Context Protocol인데, Claude Code를 외부 도구랑 연결하는 거예요.
  Playwright MCP 연결하면 브라우저 자동화를 Claude Code가 해줘요. Supabase MCP는 데이터베이스 쿼리를 대신 실행하고요. Firecrawl로 웹 스크래핑도 시키고, GitHub MCP로 PR 관리도 하고...
  가능성이 무한해요. 근데 여기서 함정이 하나 있어요.
- 13/ MCP 서버를 20~30개 설정하되, 실제로 활성화하는 건 10개 미만으로 유지하세요.
  전체 도구 수는 80개 밑으로 맞춰야 해요. 왜냐면 Claude Code가 작업할 때마다 "어떤 도구 쓸까?" 판단해야 하거든요. 도구가 너무 많으면 결정 장애 와서 느려져요.
  저는 프로젝트마다 필요한 MCP만 켜고 나머지는 비활성화해요. 그러면 속도도 빠르고 정확도도 높아져요.
- 14/ Hooks 시스템도 진짜 강력해요. 특정 이벤트가 발생할 때 자동으로 실행되는 스크립트예요.
  4가지 종류가 있어요. pre-commit은 커밋 전에 실행, post-commit은 커밋 후, pre-push는 푸시 전, post-push는 푸시 후요.
  자동 린트를 pre-commit에 걸어두면 코드 스타일 걱정 안 해도 돼요. 커밋할 때마다 알아서 포맷팅해주거든요.
- 15/ 제가 쓰는 Hook 예시 하나 줄게요. pre-commit에 테스트 실행하는 거예요.
  테스트 실패하면 커밋이 안 돼요. 그러니까 버그 있는 코드가 절대 메인 브랜치로 안 들어가죠. 팀 작업할 때 특히 유용해요.
  console.log 남아있으면 경고 띄우는 Hook도 만들었어요. 깜빡하고 로그 남기는 실수를 완전히 막아줘요.
- 16/ Skills랑 Agents도 꼭 알아야 해요. Skills는 Claude Code에 추가 능력을 부여하는 거예요.
  Planner Agent는 프로젝트 전체 구조 설계해주고, Architect는 기술 스택 추천해주고, Code Reviewer는 내 코드 리뷰해주고, TDD Guide는 테스트 작성 도와주고, Security Reviewer는 보안 취약점 찾아줘요.
  이거 설치하려면 affaan-m/everything-claude-code 레포 가세요. 거기 다 있어요.
- 17/ 명령어도 많아요. /tdd 치면 테스트 주도 개발 모드로 들어가요. RED-GREEN-IMPROVE 사이클 돌면서 테스트 먼저 짜고 코드 구현하거든요.
  /plan은 작업 계획 세우고, /e2e는 엔드투엔드 테스트 만들어주고, /code-review는 코드 리뷰 돌려줘요.
  이거 알고 나면 혼자서도 팀 개발하는 느낌 나요. 정말이에요.
- 18/ 고급 기능 하나 알려드릴게요. ultrathink라는 키워드예요.
  프롬프트에 ultrathink 넣으면 Claude Code가 더 깊게 생각해요. 복잡한 아키텍처 설계하거나 어려운 버그 잡을 때 쓰면 성공률이 확 올라가요.
  단, 시간이 좀 더 걸려요. 급할 땐 빼고, 정말 중요한 결정할 때만 ultrathink 쓰세요.
- 19/ 컨테이너 격리도 중요해요. 특히 프로젝트 여러 개 동시에 진행할 때요.
  Docker나 devcontainer 쓰면 환경이 완전히 분리돼요. A 프로젝트 Node 버전이랑 B 프로젝트 Node 버전 충돌 걱정 없어지죠.
  Claude Code도 devcontainer 지원해요. .devcontainer 폴더 만들고 설정 파일 넣으면 자동으로 컨테이너 안에서 작업해요. 환경 꼬일 일이 없어요.
- 20/ Extended Thinking이라는 기능도 있어요.
  복잡한 문제 주면 Claude Code가 스스로 여러 각도로 생각해봐요. "이렇게 하면 어떻게 되지? 저렇게 하면?" 혼자 시뮬레이션 돌리는 거죠.
  시간은 오래 걸리지만 퀄리티가 확실히 달라요. 중요한 기능 만들 때 한 번씩 써보세요.
- 21/ TDD 워크플로우 제대로 돌리는 법도 알려드릴게요. RED-GREEN-IMPROVE 사이클이에요.
  RED는 실패하는 테스트 먼저 쓰는 거예요. 아직 기능 없으니까 당연히 실패하죠. GREEN은 테스트 통과할 만큼만 코드 짜는 거예요. IMPROVE는 그 코드를 깔끔하게 리팩토링하는 거고요.
  이 순서 지키면 버그가 확 줄어들어요. 해커톤 때 이렇게 했더니 테스트 커버리지 80% 넘겼거든요.
- 22/ /tdd 명령어 실전 사용법도 보여드릴게요. "사용자 로그인 기능 TDD로 만들어줘" 이렇게 하면 돼요.
  그러면 Claude Code가 먼저 테스트 코드 작성해요. "유효한 이메일과 비밀번호 입력 시 로그인 성공" 이런 테스트요. 당연히 실패하죠.
  그다음 로그인 함수 구현하고, 테스트 통과하면 코드 정리해요. 한 사이클 끝. 이걸 계속 반복하는 거예요.
